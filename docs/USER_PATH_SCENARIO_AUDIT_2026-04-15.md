# YTVidHub 用户路径与场景价值审计（2026-04-15）

## 1) 审计目标

本审计不讨论“功能有没有”，而讨论三件事：

1. 你的路径是否贴合用户真实场景（Job To Be Done）。
2. 从输入链接到结果产出，哪里让用户产生困惑或流失。
3. 你的产品最终给用户带来的情感与意义是什么，当前是否被正确表达。

---

## 2) 当前产品已经落地解决的问题（肯定项）

从代码和路径看，你已经做成了一个可用的“字幕/转录生产流水线”：

1. **单视频与批量来源统一入口**  
   支持视频、playlist、channel 的统一解析与处理，降低手工整理成本。

2. **从“抓取”到“消费”的闭环**  
   不只下载字幕，还覆盖 AI Summary、历史回看、再次进入工作区继续处理。

3. **批量任务中断恢复能力**  
   积分不足时支持“先下载可支付部分 + 剩余任务恢复”，这是真正面向生产场景的能力。

4. **面向 AI 数据准备的结构化输出思路**  
   有 Clean TXT/多格式导出、批处理、指南内容，已经不是玩具工具，而是偏工作流产品。

---

## 3) 真实用户场景（JTBD）与当前路径

## 场景 A：研究员/开发者要快速拿到可用文本
- 任务：给我高质量字幕文本，马上可用于分析/标注/LLM。
- 当前路径：首页粘贴链接 -> workspace -> 下载或摘要 -> 导出。
- 情绪变化：焦虑（怕拿不到）-> 期待（进入 workspace）-> 若成功则“掌控感”上升。

## 场景 B：处理 playlist / 大批量 URL
- 任务：几百条 URL 自动化处理，失败可恢复，不想重来。
- 当前路径：输入 playlist/channel -> 批量检查 -> 选择下载 -> 积分不足分流至 pricing -> 回到任务续跑。
- 情绪变化：任务压力大 -> 看到进度后安心 -> 付款节点最敏感（怕断流程/怕算不清）。

## 场景 C：先理解内容再决定下载
- 任务：先看 AI 总结，判断视频是否值得深处理。
- 当前路径：首页 summary 按钮 -> workspace summary -> 按需再转下载。
- 情绪变化：信息过载 -> 先得到摘要 -> 决策负担降低。

## 4) 不符合用户需求的路径（重点问题清单）

## P0（必须先修）

1. **承诺冲突：文案说“无需账号测试”，流程却强制登录**
   - 现象：首页显示 `No account required to test`，但点击主流程动作时未登录会直接拉起登录。
   - 影响：直接伤害信任，用户会认为“被套路”，尤其新访客。
   - 证据：
     - `/src/components/landing/HeroSection.tsx:272`
     - `/src/components/landing/HeroSection.tsx:159`
   - 备注：你已有访客下载接口能力，但主流程未利用。
     - `/src/lib/api.ts:289`

2. **积分认知冲突：同一能力在不同页面价格口径不一致**
   - 现象：控制条显示 Summary 成本为 7，而 pricing 页与错误弹窗口径为 2。
   - 影响：用户对计费失去确定性，付费转化与口碑都会被伤害。
   - 证据：
     - `/src/components/hero/ControlBar.tsx:233`
     - `/src/app/[locale]/(main)/pricing/page.tsx:172`
     - `/src/hook/useSubtitleDownloader.ts:991`

## P1（高优先级）

3. **多语言预期与真实可用范围不一致**
   - 现象：系统声明支持多语言路由，但实际上当前仅首页保留多语言，非首页被中间件回退到英文。
   - 影响：语言切换后进入内页会“跳语言”，用户感知是站点不稳定或翻译失效。
   - 证据：
     - `/src/i18n/routing.ts:5`
     - `/src/middleware.ts:31`
     - `/src/components/ui/LanguageSwitcher.tsx:97`

4. **AI Summary 对 playlist 的语义处理与用户预期不一致**
   - 现象：输入 playlist 做 summary 时，自动降级为单视频并仅 toast 提示。
   - 影响：用户以为在总结“整个列表”，结果其实只处理第一条，决策会被误导。
   - 证据：
     - `/src/components/landing/HeroSection.tsx:135`
     - `/src/components/landing/HeroSection.tsx:146`

## P2（建议优化）

5. **工作区对开发者友好度偏低，影响“专业工具”信任感**
   - 现象：禁用 F12、右键复制菜单（特定区域）。
   - 影响：研究/开发人群会对产品安全性与开放性产生负面判断。
   - 证据：
     - `/src/app/[locale]/(workspace)/workspace/page.tsx:58`
     - `/src/app/[locale]/(workspace)/workspace/page.tsx:79`

6. **输入校验过于宽松，前端容易放行低质量 URL**
   - 现象：只要包含 `youtube.com`/`youtu.be` 字符串就判定有效。
   - 影响：后端报错比例高，用户会把失败归因到“产品不稳定”。
   - 证据：
     - `/src/components/workspace/UrlInput.tsx:34`

## 5) 你的产品“真正在卖什么”（不是功能，是意义）

你不是在卖“字幕下载按钮”，你在卖：

1. **确定性**：从“能不能拿到字幕”变成“稳定拿到并可复用”。
2. **掌控感**：大批量任务可视化、可恢复，减少“半夜重跑”焦虑。
3. **生产效率**：把人工碎片动作压缩成标准化流程。
4. **认知减负**：先摘要后深挖，让用户更快做出是否继续投入的判断。

换句话说，YTVidHub 的产品意义是：  
**把“视频信息处理”从手工劳动升级为可管理的生产流程。**

## 6) 情感路径（用户心理流）

1. **进入前**：焦虑（怕慢、怕失败、怕成本不透明）  
2. **输入链接后**：期待（希望系统可靠处理）  
3. **处理中**：需要安全感（进度、状态、失败可恢复）  
4. **出结果时**：获得掌控感（可下载、可复用、可继续）  
5. **付费节点**：需要公平感（计费清楚、承诺一致）

当前主要掉点在第 5 步（计费口径）和第 1~2 步（承诺与登录门槛冲突）。

---

## 7) 建议的落地优先级（30 天）

## Week 1（先修信任）
1. 统一所有页面的 Summary 积分口径（单一事实源）。  
2. 统一“游客可试用”策略：要么真正可试用，要么文案明确“登录后使用”。  
3. 在 Summary 处理 playlist 前给出显式选择：`仅首条` 或 `改走批量流程`。

## Week 2（修路径一致性）
1. 明确多语言策略文案（例如：目前仅首页多语言，工具页英文）。  
2. 语言切换后在非支持页面给出可解释反馈，而不是纯跳转感受。  
3. 输入校验升级为“结构化解析 + 失败原因可读”。

## Week 3-4（强化场景表达）
1. 首页首屏按场景给 CTA：  
   - `我要批量导出训练数据`  
   - `我要先看 AI 摘要再决定`  
   - `我要快速拿字幕文本`
2. pricing 文案由“套餐推销”改为“任务完成承诺”（例如“今天把 300 条跑完”）。
3. 增加结果页场景回流：导出后给“下一步建议”（清洗、翻译、摘要、批量续跑）。

---

## 8) 建议跟踪指标（验证这次改造是否有效）

1. 首页输入后到 workspace 的到达率。  
2. 登录弹窗出现后的继续率（而不是关闭率）。  
3. Summary 发起率 -> Summary 完成率。  
4. 批量任务从“积分不足”到“成功续跑”的转化率。  
5. 首次会话 24 小时内复访率（验证“掌控感”是否建立）。

---

## 9) 一句话结论

你的基础能力已经覆盖真实生产场景；当前最大问题不是“功能缺失”，而是**路径承诺与计费认知不一致**。  
先修信任，再放大场景价值，转化会明显提升。
